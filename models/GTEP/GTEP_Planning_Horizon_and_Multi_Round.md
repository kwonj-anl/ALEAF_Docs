# GTEP Planning Horizon and Multi-Round

This page explains how A-LEAF constructs the planning horizon for generation and transmission expansion planning, and how the optional multi-round solution process breaks that horizon into overlapping subproblems.

The descriptions in this page are based on the actual implementation in:
- [LCO_GTEP.jl](C:/ALEAF/a-leaf/src/model/LCO_GTEP.jl)

## Where The Planning Horizon Comes From
The planning horizon is controlled mainly by the `Planning Design` sheet in the simulation workbook.

The core fields are:
- `current_year_value`
- `lead_year_value`
- `final_year_value`
- `num_stages_value`
- `num_years_per_stage_value`

At runtime, `generate_networkdata_LC_GTEP` uses those values to build `network_data["planning_stages"]`.

For each planning stage `id`, the code stores:
- `stage_id = id`
- `year = current_year_value + lead_year_value + (id - 1) * num_years_per_stage_value`
- `stage_length = num_years_per_stage_value`

It also derives stage-level scaling terms such as:
- load growth multiplier
- policy year references
- time-series year references

## What The Planning Horizon Means In Practice

### `current_year_value`
This is the economic reference year for discounting and cost accounting.

It is used throughout the objective and reporting logic when A-LEAF computes present-value cost terms such as:
- generator investment cost
- transmission investment cost
- fixed O&M
- fuel and operating costs
- tax credits and penalties

### `lead_year_value`
This is the offset between the current year and the first planning stage year.

If:
- `current_year_value = 2024`
- `lead_year_value = 6`

then the first planning stage begins in 2030.

### `num_stages_value`
This is the number of planning stages in the full GTEP problem.

It determines:
- how many stage decision blocks are created
- how many stage years are exported in planning-stage reports
- how many years are considered when computing remaining cost-recovery periods

### `num_years_per_stage_value`
This is the duration of each planning stage.

It affects:
- the derived start year for each stage
- how annual costs are rolled up into each stage
- how representative-day dispatch is scaled into annual quantities

If `num_years_per_stage_value = 5`, then a four-stage problem represents four five-year blocks.

### `final_year_value`
This is the overall study endpoint used by the workbook design.

In practice, the code's stage construction is driven directly by:
- `current_year_value`
- `lead_year_value`
- `num_stages_value`
- `num_years_per_stage_value`

So `final_year_value` is best understood as a study-design reference that should be consistent with the derived last stage year.

## Example
Suppose the workbook uses:
- `current_year_value = 2024`
- `lead_year_value = 6`
- `num_stages_value = 4`
- `num_years_per_stage_value = 5`

Then A-LEAF constructs:
- Stage 1: start year 2030, length 5
- Stage 2: start year 2035, length 5
- Stage 3: start year 2040, length 5
- Stage 4: start year 2045, length 5

This means:
- investment decisions are made at the stage level
- each stage represents a five-year block
- representative-day dispatch and annualized outputs are scaled to that stage length

## How Stage Length Enters The Model
Stage length matters in several places.

### Cost accounting
The objective and reporting logic use `stage_length` to extend annual cost terms across the years represented by a stage, including:
- fixed O&M
- operating costs
- scarcity costs
- carbon cost
- policy penalty terms

### Investment recovery windows
When A-LEAF computes the remaining duration over which an investment or credit applies, it uses:
- the current stage year
- the number of remaining modeled years
- the asset cost-recovery period

This is why the code repeatedly computes values such as:
- `remaining_years`
- `payment_duration`

### Annualized reporting
The expansion summary reports include both:
- planning-stage values
- annualized cash-flow style values

Those annual outputs are generated by expanding each stage over its `stage_length`.

## Standard Single-Run Expansion
If `Planning Design["multi_round_solution_process_flag"] == false`, A-LEAF solves the full planning horizon in one expansion model.

That path uses:
- `execute_LC_GTEP_direct`

In this mode:
- all planning stages are solved together
- investment decisions are optimized jointly over the full horizon
- expansion reporting is generated once from the full solution

## Multi-Round Expansion
If `Planning Design["multi_round_solution_process_flag"] == true`, A-LEAF uses:
- `execute_LC_GTEP_direct_multi_round`

This mode solves the planning problem in sequential rounds instead of one full-horizon solve.

### Why Multi-Round Exists
The multi-round workflow is designed to make large planning problems more manageable by:
- reducing the number of stages in each solve
- allowing overlap between rounds
- carrying forward prior investment decisions
- optionally updating capacity credit inputs between rounds

## Multi-Round Control Parameters
The main workbook fields are:
- `multi_round_solution_process_flag`
- `num_stages_per_simulation_round_value`
- `num_overlaps_between_simulation_rounds_value`
- `continue_from_previous_run_flag`

### `num_stages_per_simulation_round_value`
This is the number of planning stages modeled in each round.

### `num_overlaps_between_simulation_rounds_value`
This is the number of trailing stages from the prior round that are re-included in the next round.

The overlap is used to smooth the transition between rounds and reduce end-of-horizon artifacts inside a single round.

### Derived decision-stage count
The code defines:
- `num_decision_stages_per_round = num_stages_per_simulation_round - num_overlaps_between_simulation_rounds`

This is the number of new decision stages contributed by each non-final round.

## How Round Boundaries Are Built
For each round, the code computes:
- `round_first_y`
- `round_last_y`
- `round_ids_y`
- `round_ids_y_decision`

Conceptually:
- `round_ids_y` are all stages modeled in the round
- `round_ids_y_decision` are the stages whose decisions are committed from that round

For non-final rounds:
- only the non-overlap portion is treated as the committed decision window

For the final round:
- all remaining stages in the round are treated as decision stages

## Example Multi-Round Layout
Suppose:
- total stages = 6
- stages per round = 3
- overlap = 1

Then:
- decision stages per round = 2

The rounds are:
- Round 1 models stages 1-3 and commits stages 1-2
- Round 2 models stages 3-5 and commits stages 3-4
- Round 3 models stages 5-6 and commits stages 5-6

This is the core idea of the overlapping multi-round design.

## How Decisions Carry Forward
The multi-round code stores prior decisions in:
- `recorded_investment_decisions`

That structure is passed into later rounds so that previously committed expansion decisions are reflected in the next network-data build.

This matters especially for:
- renewable capacity buildout
- storage additions
- transmission expansion state

## Relationship To RA And Capacity Credit
The multi-round workflow can optionally interact with RA.

When enabled, the code may:
- run RA after a round
- record `RA_Info` under the multi-round information structure
- apply updated ELCC / capacity credit values into the next round

The key case-level control is:
- `update_CAPCRED_in_each_round_of_Expansion_Flag`

When this is true, the next round can consume:
- `ELCC_result_applied`

for eligible technologies.

## Continuing A Prior Multi-Round Run
If:
- `continue_from_previous_run_flag == true`

the multi-round workflow checks for a prior:
- `GTEP_multi_round_info.json`

and resumes from the recorded state if that file shows the run was previously `"in progress"`.

## Multi-Round Output Artifact
The multi-round workflow writes:
- `GTEP_multi_round_info.json`

This file tracks:
- round status
- network data by committed decision year
- recorded investment decisions
- round-linked RA information when applicable

It is an internal workflow artifact rather than the main user-facing expansion report set.

## Relationship To Representative Days
The planning horizon and representative-day structure are linked but different:
- planning stages define the long-term year blocks
- representative days define the reduced operational chronology inside each stage

In multi-round mode, the round logic changes which stages are active in a given solve, but each active stage still carries its own representative-day information.

## Practical Reading Guide
When reviewing a GTEP case, check the planning horizon in this order:
1. `Planning Design` horizon fields
2. whether multi-round is enabled
3. stages per round and overlap
4. whether capacity credit is updated between rounds
5. whether continuation from a prior round file is enabled
